# 07. MySQL Router 构建读写分离

本文介绍如何在 MG R集群前端部署 MySQL Router以实现读写分离、读负载均衡，以及故障自动转移。

MySQL Router是一个轻量级的中间件，它采用多端口的方案实现读写分离以及读负载均衡，而且同时支持 mysql 和mysql x协议。

建议把 MySQL Router 部署在应用服务器上，每个应用服务器都部署一套，这样应用程序可以直接连接本机IP，连接的效率更高，而且后端数据库发生变化时，程序端也无需修改IP配置。

## 1. 部署MySQL Router

下载

```shell
cd /usr/local
wget https://cdn.mysql.com//Downloads/MySQL-Router/mysql-router-8.0.46-linux-glibc2.17-x86_64.tar.xz
tar xvf mysql-router-8.0.46-linux-glibc2.17-x86_64.tar.xz
ln -s  mysql-router-8.0.46-linux-glibc2.17-x86_64 mysqlrouter
cat >> /etc/profile << 'EOF'
export PATH=/usr/local/mysqlrouter/bin:$PATH
EOF
source /etc/profile
mysqlrouter -V
```

MySQL Router第一次启动时要先初始化：

```shell
#参数解释
# 参数 --bootstrap 表示开始初始化
# 参数 mgr@10.0.0.51:3306 是MGR服务专用账号
# --user=mysqlrouter 是运行mysqlrouter进程的系统用户名
#
[root@localhost]# mysqlrouter --bootstrap mgr@10.0.0.51:3306 --user=mysql
Please enter MySQL password for mgr:
# Bootstrapping system MySQL Router 8.0.46 (MySQL Community - GPL) instance...

- Creating account(s) (only those that are needed, if any)
- Verifying account (using it to run SQL queries that would be run by Router)
- Storing account in keyring
- Adjusting permissions of generated files
- Creating configuration /usr/local/mysql-router-8.0.46-linux-glibc2.17-x86_64/mysqlrouter.conf

# MySQL Router configured for the InnoDB Cluster 'MGR01'

After this MySQL Router has been started with the generated configuration

    $ /etc/init.d/mysqlrouter restart
or
    $ systemctl start mysqlrouter
or
    $ mysqlrouter -c /usr/local/mysql-router-8.0.46-linux-glibc2.17-x86_64/mysqlrouter.conf

InnoDB Cluster 'MGR01' can be reached by connecting to:

## MySQL Classic protocol

- Read/Write Connections: localhost:6446
- Read/Only Connections:  localhost:6447

## MySQL X protocol

- Read/Write Connections: localhost:6448
- Read/Only Connections:  localhost:6449

```

创建system配置

```shell
cat > /usr/lib/systemd/system/mysqlrouter.service <<'EOF'
[Unit]
Description=MySQL Router
Documentation=https://dev.mysql.com/doc/mysql-router/en/
After=network.target

[Service]
Type=simple
User=mysql
Group=mysql
ExecStart=/usr/local/mysqlrouter/bin/mysqlrouter -c /usr/local/mysqlrouter/mysqlrouter.conf

# 生产环境关键优化
Restart=on-failure
RestartSec=5
LimitNOFILE=65536
PrivateTmp=true

[Install]
WantedBy=multi-user.target
EOF
```

## 2. 启动mysqlrouter服务

这就初始化完毕了，按照上面的提示，直接启动 **mysqlrouter** 服务即可：

```shell
systemctl start mysqlrouter

ps -ef | grep -v grep | grep mysqlrouter
mysqlro+  6026     1  5 09:28 ?        00:00:00 /usr/bin/mysqlrouter

netstat -lntp | grep mysqlrouter
tcp        0      0 0.0.0.0:6446            0.0.0.0:*               LISTEN      6026/mysqlrouter
tcp        0      0 0.0.0.0:6447            0.0.0.0:*               LISTEN      6026/mysqlrouter
tcp        0      0 0.0.0.0:6448            0.0.0.0:*               LISTEN      6026/mysqlrouter
tcp        0      0 0.0.0.0:6449            0.0.0.0:*               LISTEN      6026/mysqlrouter
tcp        0      0 0.0.0.0:8443            0.0.0.0:*               LISTEN      6026/mysqlrouter
```

可以看到 **mysqlrouter** 服务正常启动了。

**mysqlrouter** 初始化时自动生成的配置文件是 `/usr/local/mysqlrouter/mysqlrouter.conf`，主要是关于R/W、RO不同端口的配置，例如：

```toml
[routing:bootstrap_rw]
bind_address=0.0.0.0
bind_port=6446
destinations=metadata-cache://MGR01/?role=PRIMARY
routing_strategy=first-available
protocol=classic
```

可以根据需要自行修改绑定的IP地址和端口，也可以在初始化时指定 `--conf-base-port` 选项自定义初始端口号。

## 3. 确认读写分离效果

现在，用客户端连接到6446（读写）端口，确认连接的是PRIMARY节点：

```sql
mysql -h 10.0.0.51 -P 6446 -u mgr -p
Enter password:
...
#记住下面几个 MEMBER_ID

```

```sql
mysql> select @@server_uuid;

```

## 4. 确认只读负载均衡效果

MySQL Router连接读写节点（Primary节点）默认的策略是 **first-available**，即只连接第一个可用的节点。Router连接只读节点（Secondary节点）默认的策略是 **round-robin-with-fallback**，会在各个只读节点间轮询。

保持6447端口原有的连接不退出，继续新建到6447端口的连接，查看 **server_uuid**，这时应该会发现读取到的是其他只读节点的值，因为 **mysqlrouter** 的读负载均衡机制是在几个只读节点间自动轮询。在默认的 **round-robin-with-fallback** 策略下，只有当所有只读节点都不可用时，只读请求才会打到PRIMARY节点上。

关于Router的连接策略，可以参考MySQL Router官方文档：[routing_strategy参数/选项](https://gitee.com/link?target=https%3A%2F%2Fdev.mysql.com%2Fdoc%2Fmysql-router%2F8.0%2Fen%2Fmysql-router-conf-options.html%23option_mysqlrouter_routing_strategy)

## 5. 确认故障自动转移功能

接下来模拟PRIMARY节点宕机或切换时，**mysqlrouter** 也能实现自动故障转移。

登入MGR集群任意节点：

```javascript
mysqlsh --uri mgr@10.0.0.52:3306
...
JS >  var c=dba.getCluster();
JS >  c.setPrimaryInstance('10.0.0.52:3306');   <-- 切换PRIMARY节点
```

回到前面连接6446端口的那个会话，再次查询 **server_uuid**，此时会发现连接自动断开了：

```sql

mysql> select @@server_uuid;
ERROR 2013 (HY000): Lost connection to MySQL server during query
No connection. Trying to reconnect...
Connection id:    391
Current database: *** NONE ***

+--------------------------------------+
| @@server_uuid                        |
+--------------------------------------+
| d0d2ebc6-565c-11f1-a9c8-000c29babf0e |
+--------------------------------------+
1 row in set (0.03 sec)
```

这就实现了自动故障转移。

至此，利用MySQL Router配合 MySQL 构建一套支持读写分离、读负载均衡以及故障自动转移的MGR集群就部署完毕了。

本文介绍了如何利用MySQL Router实现读写分离、读负载均衡，以及故障自动转移，利用MySQL Router可以提升应用端的透明性，后端数据库发生一些变化时，应用端无需跟着频繁变更。
