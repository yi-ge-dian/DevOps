# 1. 目标

在企业环境中，无论是安全人员、运维人员、开发人员、数据库管理人员等所有技术人员都有一个共同的职责：保障数据安全，防止数据库损坏。

1. 数据库物理损坏：磁盘、文件系统、数据文件（可以利用主从、高可用、备份+日志恢复数据）。
2. 数据库逻辑损坏：drop、truncate、delete、update（可以利用备份+日志、延时从库）。

其中对于数据库服务来说，保障数据库服务的数据安全需要考量两个重要的指标：

1. 一定要保障数据不能丢失和泄露；
2. 一定要保障数据存储服务的稳定；（业务7 * 24）。

为了保障数据信息不丢失，最好的处理方案就是做备份，甚至是做多副本备份，多区域备份；就算丢失损坏也能快速复原。

# 2. 备份恢复方式

## 2.1. 逻辑备份恢复  （50G以内） 

优势：可以更灵活的备份指定信息或恢复指定信息

缺陷：当数据量大时，备份数据效率低，恢复数据效率低

如何实现逻辑备份：

- 可以在任意时间节点在不停机不停止业务时，然后利用专业的mysqldump(MDP)逻辑备份工具进行数据备份；
- 可以在任意时间节点在不停机不停止业务时，然后利用二进制日志binlog文件实现逻辑备份数据操作；
- 可以在任意时间节点在不停机不停止业务时，然后利用主从数据库架构实现备份数据信息；

### 2.1.1. mysqldump

方式一：基础备份数据 （-A -B -F）

数据库全备：

```Bash
# 本地备份：
mysqldump -uroot -p123123 -S /tmp/mysql3306.sock -A >/tmp/all.sql

# 本地恢复：
source /tmp/all.sql
mysql -uroot -p123123 -S /tmp/mysql3306.sock </tmp/all.sql

# 远程备份：
mysqldump -uroot -p123123 -h10.0.0.51 -P3306 -A >/tmp/all.sql

# 远程恢复：
mysql > source /tmp/all.sql
mysql -uroot -p123123 -S /tmp/mysql3306.sock </tmp/all.sql
```

数据库分库备份：

```Bash
mysqldump -uroot -p123123 -S /tmp/mysql3306.sock -B oldboy >/tmp/oldboy.sql
mysqldump -uroot -p123123 -S /tmp/mysql3306.sock -B oldboy xiaoA xiaoB >/tmp/oldboy_xiaoA_xiaoB.sql

# 恢复
source /tmp/oldboy_xiaoA_xiaoB.sql
mysql -uroot -p123123 -S /tmp/mysql3306.sock </tmp/oldboy_xiaoA_xiaoB.sql
```

数据库分表备份：

```Bash
mysqldump -uroot -p123123 -S /tmp/mysql3306.sock school student teacher >/tmp/shcool.sql
# 恢复
use school;
source /tmp/shcool.sql
mysql -uroot -p123123 -S /tmp/mysql3306.sock school </tmp/shcool.sql
```

备份数据库数据后，可以切割日志信息，便于对数据信息进行全备恢复+增量恢复

```Bash
mysqldump -uroot -p123123 -S /tmp/mysql3306.sock -A -F >/tmp/all.sql
```

方式二：进阶备份数据 （--master-data=2（--source-data） --single-transaction  -R -E --triggers）

进阶参数01：--master-data=(1 2)   

-- 在备份文件中，会记录备份数据的位置点

--master-data=1  备份位置没有注释

CHANGE MASTER TO MASTER_LOG_FILE='xiaoQ_binlog.000001', MASTER_LOG_POS=540;      

--master-data=2  备份位置会有注释

-- CHANGE MASTER TO MASTER_LOG_FILE='xiaoQ_binlog.000001', MASTER_LOG_POS=540;

进阶参数02：--single-transaction  

-- 在备份文件过程中，会生成备份数据快照信息（避免了数据库服务出现锁库的情况）

进阶参数03：-R -E --triggers （恢复开发相关的数据信息）

-R  备份数据库中存储过程和函数信息  

https://www.runoob.com/w3cnote/mysql-stored-procedure.html

-E  备份数据库中事件相关数据信息  

https://blog.csdn.net/weixin_30831261/article/details/113159414    

--triggers  备份数据库中触发器数据信息 

https://www.php.cn/faq/492867.html

参数实践用法

```Bash
mysqldump -uroot -p123123 -h10.0.0.51 -P3306 -A  --source-data=2  --single-transaction -R -E --triggers
```

## 2.2. 物理备份恢复  （50G以上）

优势：可以更快速备份数据信息（数据量大） 

缺陷：进行指定数据恢复时，不够灵活

如何实现物理备份：

- 可以在某个特定时间点停机或停止业务访问，然后利用cp和tar命令将物理数据文件备份或打包；
- 可以在任意时间节点在不停机不停止业务时，然后利用专业的xtrabackup(`P`ercona `X`tra`b`ac`k`up)热备工具进行数据库数据备份；

### 2.2.1. xbk

下载物理备份软件工具：

https://www.percona.com/downloads/ 

xbk工具使用兼容性说明：

对于数据库 8.0.20版本，   需要使用 PXB 8.0.12+以上版本

对于数据库 8.0.11~8.0.19，需要使用 PXB 8.0    正式版本；

PXB 8.0 只能备份 MySQL 8.0 版本数据，不能备份低版本数据信息；

如果想备份数据库服务低版本程序数据，需要下载使用 PXB 2.4 版本；

安装 xbk 软件程序

```Bash
yum install -y percona-xtrabackup-80-8.0.13-1.el7.x86_64.rpm
```

备份逻辑（支持增量备份数据）：

1. 在数据库服务运行期间，通过拷贝数据文件（实质拷贝的是数据页），进而实现数据备份目的；
2. 在进行数据文件拷贝的同时，会将备份期间的变化 redo 日志信息同时进行备份（拷贝）；

恢复逻辑：

1. 在进行数据恢复时，模拟了InnoDB Crash recovery（CR）的运行过程，需要将备份进行处理，才能进行数据恢复；
2. 在进行数据恢复时，对于备份进行处理操作，特指的就是前滚操作（redo）和回滚操作（undo），从而解决数据恢复一致性问题；

全量备份数据过程：

```Bash
mkdir -pv /data/backup/full     # 备份数据目录必须是空目录
xtrabackup --defaults-file=/etc/my.cnf --host=10.0.0.51 --user=root --password=123123 --port=3306 --backup  --target-dir=/data/backup/full
```

全量恢复数据过程：

```Bash
# 停止数据库服务
systemctl stop mysqld
# 将数据库数据目录中信息删除
rm -rf /data/3306/data/*
rm -rf /data/3306/log/*
# 读取增量数据信息，恢复内存数据
xtrabackup --prepare --target-dir=/data/backup/full
# 读取全量数据信息，恢复磁盘数据  
xtrabackup --copy-back  --target-dir=/data/backup/full
chown -R mysql.mysql /data/3306/ 
```

增量备份数据过程:

```Bash
# 创建备份数据目录
mkdir /data/backup/full -p 
mkdir /data/backup/week02 -p 
mkdir /data/backup/week03 -p

# 完成数据库数据信息全备
xtrabackup --defaults-file=/etc/my.cnf --host=10.0.0.51 --user=root --password=123123 --port=3306 --backup --parallel=4 --target-dir=/data/backup/full
# 进行增量备份
# do something
xtrabackup --defaults-file=/etc/my.cnf --host=10.0.0.51 --user=root --password=123123 --port=3306 --backup --parallel=4 --target-dir=/data/backup/week02 --incremental-basedir=/data/backup/full
# do something
xtrabackup --defaults-file=/etc/my.cnf --host=10.0.0.51 --user=root --password=123123 --port=3306 --backup --parallel=4 --target-dir=/data/backup/week03 --incremental-basedir=/data/backup/week02
```

增量恢复数据过程：

```Bash
# 停止数据库服务
systemctl stop mysqld
# 将数据库数据目录中信息删除
rm -rf /data/3306/data/*
rm -rf /data/3306/log/*
# 数据恢复过程
xtrabackup --prepare  --apply-log-only  --target-dir=/data/backup/full
xtrabackup --prepare  --apply-log-only  --target-dir=/data/backup/full --incremental-dir=/data/backup/week02
xtrabackup --prepare  --apply-log-only  --target-dir=/data/backup/full --incremental-dir=/data/backup/week03
# 将以上整合全量+增量数据 恢复到内存区域
xtrabackup --prepare  --target-dir=/data/backup/full
# 将以上整合全量+增量数据 恢复到磁盘区域
xtrabackup  --copy-back  --target-dir=/data/backup/full
chown -R mysql.mysql /data/3306/
```

# 3. 企业中如何修复数据

## 3.1. 日志文件被清理过，可能建库语句所在日志已经丢失，如何修复数据

drop database xiaoA;  

binlog10 11 12

解决方案：

A 计划：最近一次全备+全备之后，误删除之前所有 binlog，进行一同恢复；（`全备数据+增量数据`）

B 计划：利用延时从库，进行数据恢复；

按照 A 计划进行修复数据

```SQL
-- 步骤一：模拟数据信息 
create database test01;
use test01; 
create table t1 (id int,name char(10));
flush logs;
insert into t1 values (1,'xiaoA');
insert into t1 values (2,'xiaoB');

-- 步骤二：进行数据库备份操作
mysqldump -uroot -p123123 -S /tmp/mysql3306.sock --source-data=2 --single-transaction -B test01 >/tmp/test01.sql

-- 步骤三：添加增量数据信息
use test01;
insert into  t1 values (3,'xiaoC');
insert into  t1 values (4,'xiaoC');

-- 步骤四：模拟数据损坏 
drop database test01;

-- 步骤五：恢复数据信息
mysql -uroot -p123123 -S /tmp/mysql3306.sock </tmp/test01.sql  -- 位置点信息？？ MASTER_LOG_FILE='xiaoQ_binlog.000010', MASTER_LOG_POS=734;
mysqlbinlog --start-position=734 --stop-position=1312 xiaoQ_binlog.000010 >/tmp/add.sql 
mysql -uroot -p123123 -S /tmp/mysql3306.sock </tmp/add.sql
```

## 3.2. 所需日志跨越多个文件，如何进行日志信息的截取；

解决方案： 

A 计划：只有 position 号的方式，可以进行分段截取，进行分段恢复数据；

B 计划：根据 Datatime 时间信息方式，可能会出现准确性不高的情况（因为每一秒可能有多个事件产生）；

C 计划：启用 GTID（全局事务 ID）方式，无论跨越多少个日志文件，每个事务操作的事件 ID 信息都是唯一且递增的（5.6+引入）；

什么是 GTID？

全局事务 ID （global transaction ID），对数据库中每一个操作事务信息设置一个编号

create database test01  -- DDL --  事件 01  -- binlog001 

create table    t1      -- DDL --  事件 02  -- binlog002   

insert into t1          

insert into t1 

insert into t1 

commit                  -- DML --  事件 03  -- binlog003

drop databas test01     -- DDL --  事件 04  -- binlog003

过滤方式 

1）过滤和库有关语句

mysqlbinlog binlog001 binlog002 binlog003  -- create database test01  -- 事件 01

​                                              create table    t1      -- 事件 02 

mysqlbinlog 解码参数  binlog001 binlog002 binlog003         insert into t1 表有关 dml 语句  -- 事件 04      

​                                                                                          

事件 01  事件 02  事件 03   事件 04

mysqlbinlog gitd 事件范围信息 1:4  binlog001 binlog002 binlog003  >binlog.sql 

gtid 编号：server_uuid:N

server_uuid: 表示数据库初始化启动之后，自动生成的随机数信息（唯一的）

N: 表示第几个相关的事务或事件信息，会不断进行自增

```SQL
开启 gtid 功能：
select @@gtid_mode;                  -- 开启此功能 on 
select @@enforce_gtid_consistency;   -- 开启此功能 on  

实践操作：
步骤一：模拟创建测试数据
create database test05;                   4    -- xiaoQ_binlog.000013
create table t1 (id int,name char(10));   5  
flush logs;   
insert into t1 values (1,'xiaoA');        6    -- xiaoQ_binlog.000014
flush logs;                                    
insert into t1 values (2,'xiaoB');        7    -- xiaoQ_binlog.000015

步骤二：模拟删除数据信息                 
drop database test05;                     10    -- xiaoQ_binlog.000015

步骤三：修复数据信息
mysqlbinlog --skip-gtids --include-gtids='31a042c9-345f-11ef-87bc-000c29c19eee:4-7' xiaoQ_binlog.000013  xiaoQ_binlog.000014 xiaoQ_binlog.000015 >/tmp/gtid.sql
 
--exclude-gtids='7afe4f8c-5e36-11ed-b083-000c29d44f34:5-9'  --exclude-gtids='7afe4f8c-5e36-11ed-b083-000c29d44f34:10-13'
```

## 3.3. 如何从日志文件中恢复单库、单表、或者部分行数据信息；  

drop database xiaoA;  binlog0001  所有数据库 所有表信息

drop table xiaoA      binlog0001  

delete from t1 where xxx 

A 计划：可以利用命令单独截取某个数据库的日志信息；`mysqlbinlog -d world  xxx > xxxx`

B 计划：可以借助第三方工具实现单表或部分数据恢复；`binlog2sql（python）` 过滤指定表数据或过滤指定表的部分数据；

delete 语句   -- 数据闪回  -- insert

insert 语句   -- 数据闪回  -- delete     

update 语句   -- 数据闪回  -- update t1 set=a=1 a=2  update t1 set=a=2 a=1
