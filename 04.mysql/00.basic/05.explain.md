# 1. 数据库执行计划概念

在介绍数据库服务程序运行逻辑时，在 SQL 层处理 SQL 语句时，会先通过解析器生成解析树（对应多种可行的处理方案）；

然后利用优化器基于解析树生成**最优的执行计划**，再按照这个最优执行计划执行 SQL 语句；

作为数据库管理员，可以在某条 SQL 语句执行前，提前提取并分析该语句对应的执行计划，从而大致判断语句的执行行为、了解执行效率与效果；

可以简单理解为：执行计划就是数据库选择的最优 SQL 执行方案，用于描述 SQL 语句的具体执行方式。

# 2. 数据库执行计划获取

使用命令 explain/desc

```SQL
desc select * from world.city where countrycode='CHN';
+----+-------------+-------+------------+------+---------------+-------------+---------+-------+------+----------+-----------------------+
| id | select_type | table | partitions | type | possible_keys | key         | key_len | ref   | rows | filtered | Extra                 |
+----+-------------+-------+------------+------+---------------+-------------+---------+-------+------+----------+-----------------------+
|  1 | SIMPLE      | city  | NULL       | ref  | CountryCode   | CountryCode | 12      | const |  363 |   100.00 | Using index condition |
+----+-------------+-------+------------+------+---------------+-------------+---------+-------+------+----------+-----------------------+
```

各个字段的解释：

| 字段名        | 含义说明                                                     |
| ------------- | ------------------------------------------------------------ |
| id            | 执行计划编号，多表查询时会显示多行执行计划结果               |
| select_type   | 语句查询类型，simple 表示简单查询                            |
| partitions    | 表分区后，显示从哪个分区调取数据信息                         |
| type          | 索引应用类型，判断是否使用索引、是否合理使用索引（如 ALL、const）all：全表扫描、index：索引全扫描、range：索引范围扫描、ref：非唯一索引等值匹配、eq_ref：唯一索引等值匹配、const：常数匹配【扫描效果依次递增】 |
| possible_keys | 显示查询过程中**可能被使用**的所有索引信息                   |
| key           | 显示查询过程中**最终实际使用**的索引列信息                   |
| key_len       | 索引覆盖长度，用于评估联合索引使用情况（全覆盖 / 部分覆盖 / 全不覆盖） |
| rows          | 查询过程中需要扫描的表行数据量（估算值），越少越好           |
| filtered      | 数据匹配过滤比例，计算公式：输出数据行数 / 扫描行数 = xxx%   |
| Extra         | 额外提示信息，显示是否使用了索引优化、查询算法等功能         |

# 4. 索引覆盖长度

在执行计划列中，key_len 主要用于判断联合索引覆盖长度（字节），当覆盖长度越长，就表示匹配度更高，回表查询的次数越少；

到底联合索引被覆盖了多少，是可以通过 key_len 计算出来。

key_len 长度计算方法：

- 创建表时，字符集有关：utf8mb3、utf8mb4
- 索引列数据类型有关：字符类型、数值类型
- 索引列是否设置为空有关

| 数据类型    | 字符集  | 计算结果                                                   |
| ----------- | ------- | ---------------------------------------------------------- |
| char(10)    | utf8mb4 | 最大预留长度=4*10=40 10                                    |
|             | utf8    | 最大预留长度=3*10=30                                       |
| varchar(10) | utf8mb4 | 最大预留长度=4*10=40 + 2字节 =42 (1-2字节存储字符长度信息) |
|             | utf8    | 最大预留长度=3*10=30 + 2字节 =32 (1-2字节存储字符长度信息) |
| tinyint     | N/A     | 最大预留长度=1（大约3位数）                                |
| int         | N/A     | 最大预留长度=4（大约10位数）                               |
| bigint      | N/A     | 最大预留长度=8（大约20位数）                               |
| not null    | N/A     | 在没有设置 not null 时，在以上情况计算结果再+1             |

## 4.1. key_len 计算练习

```SQL
use test;
create table keylen (
    id int not null primary key auto_increment,
    k1 int not null,           --            4
    k2 char(20),               -- 4x20+1 =   81
    k3 varchar(30) not null,   -- 4x30+2 =   122
    k4 varchar(10)             -- 4*10+2+1 = 43
) charset=utf8mb4;   -- 4

-- 4 +81 + 122 + 43 =  250
```

设置表中列索引信息

```SQL
alter table keylen add index idx(k1,k2,k3,k4);
insert into keylen values (1,101,'oldboy','linux','xiaoQ');
desc select * from keylen where k1=101 and k2='oldboy' and k3='linux' and k4='xiaoQ';
-- key_len = 250
```

# 6. 数据库索引应用扩展

## 6.1. 排序优化

Extar列表示额外的情况或额外的信息说明，其中重点需要关注点信息为：filesort 表示涉及到额外排序操作，将严重浪费CPU资源；

哪些查询数据信息排序过程

- 利用order by 可以显示进行排序
- 利用group by 可以隐式进行排序
- 利用DISTINCT 可以隐式进行排序 

PS：建立索引结构时，会对索引列信息进行排序处理

没设置联合索引

```SQL
mysql> desc select * from t100w where k1='At' order by num;
+----+-------------+-------+------------+------+---------------+------+---------+-------+------+----------+---------------------------------------+
| id | select_type | table | partitions | type | possible_keys | key  | key_len | ref   | rows | filtered | Extra                                 |
+----+-------------+-------+------------+------+---------------+------+---------+-------+------+----------+---------------------------------------+
|  1 | SIMPLE      | t100w | NULL       | ref  | k1            | k1   | 9       | const | 1056 |   100.00 | Using index condition; Using filesort |
+----+-------------+-------+------------+------+---------------+------+---------+-------+------+----------+---------------------------------------+
```

需要创建联合索引

```SQL
alter table t100w add index xiaoQ(k1,num);
mysql> desc select * from t100w where k1='At' order by num;
```

## 6.2. ICP-索引下推

减少IO消耗 减少回表次数。简单理解：可以将数据库服务层无法过滤的数据信息，交给引擎层进行过滤处理。

```Bash
select * from t100w where k1='qj' and k2 like '%v%;  
```

区别

没有索引下推 (ICP) 的流程

1. 存储引擎层：利用联合索引 `(k1, k2)`，快速定位到所有 `k1='qj'` 的索引项。
2. 存储引擎层：对于每一个满足 `k1='qj'` 的索引项，无条件地执行回表操作，回到主键索引（聚簇索引）中获取完整的行数据（因为查询是 `SELECT *`）。
3. 服务器层：接收存储引擎返回的完整行数据，再根据 `k2 LIKE '%v%'` 这个条件进行过滤，最终返回满足所有条件的数据。

问题所在：在这个过程中，即使很多行的 `k2` 字段并不包含 'v'，存储引擎也已经为它们执行了耗时的回表操作。这些回表操作都是不必要的 I/O 开销。

开启索引下推 (ICP) 后的流程

1. 存储引擎层：同样利用联合索引 `(k1, k2)`，快速定位到所有 `k1='qj'` 的索引项。
2. 存储引擎层：关键步骤！ 此时，MySQL 服务器会将 `k2 LIKE '%v%'` 这个条件也“下推”给存储引擎。存储引擎在遍历 `(k1, k2)` 索引时，会直接利用索引中已有的 `k2` 值，对 `k2 LIKE '%v%'` 条件进行判断和过滤。
3. 存储引擎层：只有那些同时满足 `k1='qj'` 并且 `k2` 包含 'v' 的索引项，才会触发回表操作，去获取完整的行数据。
4. 服务器层：接收存储引擎返回的、已经经过初步筛选的完整行数据。

开启

```Bash
show variables like '%switch%'
set global optimizer_switch='index_condition_pushdown=off';
set global optimizer_switch='index_condition_pushdown=on';

[mysqld]
optimizer_switch='index_condition_pushdown=on'
```

## 6.3. MRR-多范围读取操作

 可以减少回表过程中IO消耗，利用辅助索引信息，获取到大量数据，对应获取大量主键信息，会进行回表查询获取数据，MRR 通过把「随机磁盘读」，转化为「顺序磁盘读」，从而提高了索引查询的性能。

```Bash
# 在当前会话（Session）中启用 MRR 优化功能。
mysql > set optimizer_switch='mrr=on';

# 使用这个，在全局范围内（Global）设置 MRR 的启用策略为“基于成本”的模式。
mysql > set global optimizer_switch='mrr_cost_based=on';


[mysqld]
optimizer_switch='mrr=on,mrr_cost_based=on'
```

## 6.4. AHI - 自适应 hash 索引

InnoDB 的默认索引结构是 B+ 树。B+ 树非常适合范围查询（如 `>`, `<`, `BETWEEN`），但在做精确查找（如 `WHERE id = 100`）时，需要一层一层地遍历树节点，这会产生一定的 CPU 和 I/O 开销。

AHI 的作用就是：

InnoDB 会自动监控数据页的访问模式。如果发现某个数据页被频繁地通过等值查询访问，InnoDB 就会在内存中（Buffer Pool 内）为这个数据页自动建立一个哈希索引。

下次再查这个数据时，MySQL 就不需要遍历 B+ 树了，而是直接通过哈希算法（O(1) 时间复杂度）定位到内存中的数据页，速度极快。

```Bash
show variables like 'innodb_adaptive_hash_index';
+----------------------------+-------+
| Variable_name              | Value |
+----------------------------+-------+
| innodb_adaptive_hash_index | ON    |
+----------------------------+-------+

SET GLOBAL innodb_adaptive_hash_index = ON;
SET GLOBAL innodb_adaptive_hash_index = OFF;
```

## 6.5. change buffer-特殊内存

change buffer 优化设置 -- 特殊的内存存储空间 

当修改二级索引（Secondary Index）且该索引页不在内存中时，InnoDB 不会立即从磁盘读取该页，而是将修改操作“暂存”起来，留待以后合并。

在 InnoDB 中，更新一条数据通常涉及更新主键索引（聚簇索引）和二级索引。

- 主键索引：通常更新频率高，且数据页大概率已经在内存（Buffer Pool）中，所以直接更新即可。
- 二级索引：更新往往是随机的。如果要更新的二级索引页不在内存中，传统的做法是：
   - 从磁盘读取该索引页到内存（随机 I/O，非常慢）。
   - 在内存中修改该页。
   - 标记为脏页，等待刷盘。

Change Buffer 的做法是：

如果二级索引页不在内存中，InnoDB 不去磁盘读它，而是直接把“我要修改这个页”这个动作记录下来，存入 Change Buffer。

- 收益：省去了最耗时的“从磁盘读取索引页”这一步（随机读）。
- 后续：等到下次这个页被读取到内存，或者后台线程空闲时，再将 Change Buffer 里的修改“合并”（Merge）进去。

# 7. 应用索引信息时出现的失效情况

1. 查询数据语句用法错误
2. 在查询条件过程中，使用了函数信息在索引列上，或者对索引进行了运算（+ - * / ！等）

```Bash
select * from test where id-1=9; 
```

1. 特殊函数调用，隐式转换函数（130 数值 -- 130 字符串 ）
2. 在查询数据过程中，应用了特殊条件匹配方式：<> , not in , like "%_"  ，< ， >。输出结果集占比数据量更多 ，自动将以上匹配方式查询转换为 all 全表扫描。
3. 禁用了索引功能。

```SQL
alter table city alter index population invisible;  -- 禁用索引 
alter table city alter index population visible;    -- 激活索引
```
